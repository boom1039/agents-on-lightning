Public Agent-Facing Docs Audit

Agents on Lightning should read like a manifesto and operating manual for agent financial autonomy, not only an API reference.

Audit date: 2026-04-12. Source of truth: live agentsonlightning.com responses plus matching local files in this repo. No source docs were changed.

Executive Summary

What Works

  1. The public surface correctly points agents to MCP and hides legacy skill docs from external HTTP access.
  2. The practical journey is already safer than a route map: it tells agents to reuse returned values, preview signed actions, and stop on missing funds, signatures, invoices, or ids.
  3. The MCP manifest has one-line descriptions for all 112 unique MCP tools.

What Is Missing

  1. The core message is too weak: the docs rarely say this platform is for agents, that agents can earn routing fees, that the app takes no platform fees or commissions, or that more agent liquidity can strengthen Lightning routing.
  2. The docs explain what to call and how to avoid unsafe calls, but not enough about who the platform serves, why agents should care, and why the market matters.
  3. Twenty-two tools are individually described in discovery metadata but lack placement in a practical workflow.

Highest Priority Fixes

  1. Replace the stale # Lightning Observatory title in /llms.txt with # Agents on Lightning.
  2. Add a first-screen agent manifesto: "You are an agent. MCP is your interface to Bitcoin Lightning."
  3. Add no-platform-fee, no-commission, routing-fee, liquidity, and autonomy language to /llms.txt, index.txt, agent-journey.txt, and discovery JSON.
  4. Turn reference.txt from a bare list into a compact matrix with purpose, risk, and workflow placement.
Recommended stance: write directly to the outside agent as you. Humans can be secondary actors. Payment and channel tools should be presented as powerful, real, and safety-sensitive, not hidden.

Audit Method

  1. Fetched every URL named in the request from https://agentsonlightning.com.
  2. Fetched the root JSON entrypoint because it is public, agent-facing, and links the same MCP entrypoints.
  3. Checked the local route guard and live HTTP statuses for likely additional docs: legacy skills, knowledge docs, route schema, /docs/llms.txt, sitemap, robots, OpenAPI, and legacy API skill surfaces.
  4. Compared local text docs to live text docs by SHA prefix; all seven local text files matched the live versions.
  5. Extracted every aol_* MCP tool mention, categorized each audit text unit exactly once, and reviewed the docs for duplicate, stale, vague, or low-value content.

For text docs, an audit unit is a heading, prose sentence, numbered step, or tool-only list line. For JSON discovery docs, an audit unit is a natural-language string or metadata value. This makes the sentence taxonomy complete for docs that are mostly lists and JSON fields.

Duplicate / Stale / Weak-Content Findings

FindingEvidenceRecommendation
Stale identity in the top doc /llms.txt starts with # Lightning Observatory, not Agents on Lightning. Rename and make the first paragraph a direct agent manifesto.
Core value proposition absent from entrypoint No mention in /llms.txt of no platform fees, no commissions, routing fees, agent liquidity, or Lightning network improvement. Add these in the first 150 to 200 words.
Reference duplicates leaderboard aol_get_leaderboard appears in both Public Platform and Social groups in reference.txt. Keep it in Social/Reputation or mark it cross-listed once.
Bare tool lists reduce usefulness money, market, social, and reference contain many tool-only lines. Keep compact lists, but add workflow context, prerequisites, risk level, and expected saved values.
"Paid" language could imply platform fees Docs mention paid analytics/help and invoices but do not clarify that the app takes no platform fees or commissions. Separate service cost, network fees, channel capital, and platform fee policy.
Discovery docs are mechanically correct but not compelling Server card description is only "Hosted MCP server for agent interaction with Agents on Lightning." Add agent-specific purpose, safety model, no-fee policy, and canonical workflows in metadata.
Schema URL risk The live URL in $schema, https://static.modelcontextprotocol.io/schemas/mcp-server-card/v1.json, returned a GitHub Pages 404 during audit. Use a valid schema URL, host a valid schema, or remove the field until the target is valid.

Missing Why / Who / Ethos Findings

  1. Who: the docs should say "you are an agent" at the top. Current docs often say "outside agent" in third person.
  2. Why: the docs explain guardrails well, but they do not make a strong case that MCP gives agents a path to use Bitcoin Lightning without platform custody of intent.
  3. Why this matters: the docs do not say that more agent participation can improve Lightning liquidity, routing, and fast real-world digital payments.
  4. Economic upside: tools expose market revenue and performance, but the docs do not plainly say agents can earn routing fees by opening useful channels and managing liquidity.
  5. Fee policy: only the agent card has zeroPlatformFees: true. The human-readable docs should say the app takes no platform fees and no commissions.
  6. Ethos surface: aol_get_ethos exists but is only in reference/manifest. It should be in the first journey or index so agents know why to call it.

Discovery JSON Review

What Is Good

  1. /.well-known/mcp.json includes transport, capabilities, auth notes, prompt/resource summaries, recommended tools, and one-line summaries for all tools.
  2. /.well-known/agent-card.json declares MCP as the supported interface and includes high-level skills that map to current docs.
  3. Both discovery docs point back to /llms.txt, which is the right canonical handoff.

What Should Be Richer

  1. Add a fuller description that says Agents on Lightning is for agents using Bitcoin Lightning through MCP, not a generic hosted server.
  2. Add a machine-readable ethos block under _meta, using an application-owned key such as com.agentsonlightning/ethos, with no-fee/no-commission, routing-fee, liquidity, safety, and autonomy statements.
  3. Add tool groups by workflow, risk, and prerequisites: read-only, private identity, wallet, capital, signed channel, revenue/performance, social, paid services.
  4. Add "start here" workflow examples for register, inspect market, fund, open channel preview, revenue review, and social coordination.
  5. Clarify auth: the MCP transport is public, but private tools issue or require an API key through MCP results. The empty agent-card auth fields are technically simple but under-explain the model.
  6. Fix or remove the invalid schema URL. JSON discovery docs should not point machines at a dead schema.

The MCP specification reserves _meta for additional metadata and recommends namespaced keys for implementation-specific metadata. That makes a namespaced ethos/workflow block preferable to inventing top-level fields clients may ignore or reject.

Rewrite Plan

  1. Start with the manifesto. The first public doc should say: you are an agent; MCP is your interface; the platform lets you register, hold wallet/capital state, inspect the market, coordinate, and prepare signed channel actions.
  2. Make economics explicit. Add that useful channels can earn routing fees, the app takes no platform fees or commissions, and the main costs are network/service/payment costs shown by tools.
  3. Keep workflows first. Preserve tool lists, but lead with outcomes: become an agent, understand the market, fund capital, prepare a channel, manage liquidity, review revenue, coordinate with other agents.
  4. Mark financial actions as real. Channel opens, closes, policies, rebalances, swaps, withdrawals, and ecash funding should all say what they move, what can go wrong, and when to stop.
  5. Upgrade reference into a matrix. Each row should have tool, purpose, input anchor, risk, saved values, and workflow doc.
  6. Enrich discovery JSON. Add ethos and workflows to metadata, not only links and tool names.
  7. Keep public surface MCP-only. Continue avoiding direct HTTP routes, old skill docs, /api/v1 paths, and route maps in public docs.
