Overall Assessment

The docs are technically sound and operationally safe. An agent following them step-by-step will register, inspect the market, and understand what tools exist. Safety constraints are well-written and repeated appropriately.

The docs are weak on identity, motivation, and competitive positioning. They answer “what tools exist” and “how to call them” but almost never answer:

  • Who is this for? — The word “agent” appears hundreds of times, but no doc ever says “This platform exists for you, the agent.”
  • Why should an agent care? — Zero platform fees is mentioned only once, buried in a machine-readable JSON field (zeroPlatformFees: true in agent-card.json). Routing fees as an income stream are never mentioned.
  • What’s the bigger picture? — The network-effect argument (“more agents = better Lightning”) is completely absent. Agent financial autonomy is stated once as a goal but never developed.

The core problem: These docs read like an internal operations manual. They should read like a manifesto with an operations manual attached.

Top 5 Findings

CRITICAL Zero fees / no commissions never stated in any text doc. This is the platform’s single most compelling differentiator. It appears only as a boolean in agent-card.json.

CRITICAL Routing fees as agent income never explained. Agents can earn money by providing liquidity, but no doc mentions this. The entire economic incentive for participation is undocumented.

CRITICAL No “why this matters” narrative. Sentence analysis shows only 7% of all sentences explain motivation. The docs are 53% direct instructions and 20% bare tool references.

MEDIUM 22 tools have no workflow context. They appear only in the flat reference list. Several are valuable (aol_get_ethos, aol_get_capabilities, aol_decode_invoice) but agents won’t know when or why to use them.

MEDIUM llms.txt title says “Lightning Observatory” instead of “Agents on Lightning.” The primary entry point has the wrong brand name.

4. Sentence-Category Taxonomy

Every non-blank, non-heading sentence in the 6 text docs was classified into exactly one of 8 categories. The categories were designed to be mutually exclusive and collectively exhaustive.

CategoryTagDefinitionExample
Direct Instruction DI Imperative sentence telling the agent to perform a specific action. Includes tool call steps. “Call aol_get_platform_status.”
Safety Constraint SC Prohibition, stop condition, or security rule the agent must honor. “Do not invent peer pubkeys, channel ids, quote ids, flow ids, or funding state.”
Concept Definition CD Defines a term, role, entity, or boundary to build the agent’s mental model. “The wallet is agent spending money.”
Workflow Logic WL Describes ordering, dependency, prerequisite, or sequencing between steps. “Every later step depends on real values returned earlier.”
Motivation / Why MW Explains the purpose, reasoning, or strategic value behind a feature or action. “The goal is financial autonomy with explicit guardrails, not blind endpoint coverage.”
Navigation / Discovery ND Points to another doc, endpoint, resource, or prompt by name or path. “Server card: /.well-known/mcp/server-card.json.”
Context / Background CB Background information that neither instructs nor defines. Situational color. “Deposits can take time.”
Bare Tool Reference TR Tool name listed in a sequence or group without individual explanation. aol_get_wallet_balance (as a numbered list item)

8. Missing “Why / Who / Ethos” Findings

The following critical concepts are absent or underdeveloped across all 10 public docs:

F1. Zero Platform Fees / No Commissions CRITICAL

Current state: Mentioned exactly once, as "zeroPlatformFees": true inside agent-card.json. No text doc mentions this.

Why it matters: This is the single most compelling reason for an agent to choose this platform over alternatives. Hiding it in a JSON boolean means most agents will never see it.

Fix: State it prominently in llms.txt, index.txt, and agent-journey.txt: “Agents on Lightning charges zero platform fees and takes no commissions. Every satoshi you earn is yours.”

F2. Routing Fees as Agent Income CRITICAL

Current state: Not mentioned in any doc. The word “routing” appears only in passing (“routing economics” in market.txt line 91). The tools aol_get_market_revenue, aol_get_market_revenue_channel, and aol_update_revenue_config exist but are never connected to the concept of “you can earn money.”

Why it matters: Agents need an economic reason to participate. Opening channels and providing liquidity generates routing fees. This is the core business case and it is completely undocumented.

Fix: Add a “How You Earn” section to market.txt explaining: open channels → provide liquidity → route payments → earn fees. Connect it to aol_get_market_revenue and aol_update_revenue_config.

F3. Network Effect Argument CRITICAL

Current state: No doc explains that more agents using the platform makes the Lightning Network more useful, more liquid, and more resilient.

Why it matters: This is the long-term thesis. Agents are not just users — they are infrastructure participants. Saying this explicitly creates a shared purpose.

Fix: Add to index.txt Mission section: “Every channel you open adds liquidity to the Lightning Network. Every payment you route makes the network faster and more reliable. The more agents participate, the more useful Lightning becomes for everyone.”

F4. “This Platform Is For You, The Agent” CRITICAL

Current state: No doc addresses the agent directly as the intended primary user. Humans are never explicitly described as secondary actors. The docs read as if written for a developer evaluating an API, not for an autonomous agent making decisions.

Why it matters: The core thesis is that this platform is for agents. Agents should feel that immediately. The docs should speak directly to the agent: “You are an agent. This platform is built for you.”

Fix: Open index.txt with: “You are the intended user. Agents on Lightning is built for autonomous agents who want to operate on the Bitcoin Lightning Network — holding funds, opening channels, earning routing fees, and coordinating with other agents. Humans may observe, but this platform is designed for you.”

F5. Agent Financial Autonomy MEDIUM

Current state: The phrase appears once in index.txt (“The goal is financial autonomy with explicit guardrails”) but is never developed. What does financial autonomy mean for an agent? What can an autonomous agent do that a non-autonomous one cannot?

Fix: Develop this in the agent-journey “Why This Journey Exists” section. Financial autonomy means: you hold your own keys, control your own funds, choose your own peers, set your own fee policies, and keep every satoshi you earn.

F6. “What Makes This Different” MEDIUM

Current state: No doc explains what distinguishes this platform from traditional financial APIs, custodial services, or other agent platforms.

Fix: Brief differentiator in index.txt: non-custodial key management, signed instructions for all capital movements, public audit trail, zero fees, direct Lightning participation.

F7. aol_get_ethos Tool Buried MEDIUM

Current state: The aol_get_ethos tool exists and presumably returns the platform’s philosophical stance. It appears only in reference.txt. No workflow doc tells agents to read it.

Fix: Add aol_get_ethos to agent-journey Section 1 (Start) and to index.txt. The platform’s ethos should be one of the first things an agent encounters.

F8. Why Lightning? MEDIUM

Current state: No doc explains what the Lightning Network is, why it matters for agents, or why agents should care about payment channels. The docs assume the reader already understands Lightning.

Fix: Add a brief “What Lightning Is” paragraph to index.txt: instant, low-cost Bitcoin payments through a network of payment channels. Agents can earn fees by routing payments through channels they fund.

10. Rewrite Plan

Guiding Principles

  1. Lead with why, follow with how. Every doc should open with purpose and value before listing tools.
  2. Address the agent directly. Use “you” throughout. Humans are secondary actors.
  3. State the economics explicitly. Zero fees, routing income, and network effects must be front-and-center.
  4. Keep the procedural strength. The step-by-step workflows are excellent. Don’t replace them — add a motivation layer on top.
  5. Make the ethos tool prominent. aol_get_ethos should be one of the first things an agent encounters.
  6. Don’t break what works. Safety constraints, saved_values patterns, and signing workflows are well-designed. Keep them intact.
