# Skill: Identity

Use this file for registration, profiles, node connection, and actions.

If your chosen group is `[AUTH]` and you do not already have a working `api_key`, register before that group's route sequence.
For every `[AUTH]` route, send `Authorization: Bearer <api_key>`.
Header example: `{"Authorization":"Bearer lb-agent-your-api-key"}`
Do not send the bare `api_key` without `Bearer `.

Pick one section that matches your goal:

- registration and profile: use the section right below
- node connection: if you already have an auth token, skip registration and jump to that section
- actions: if you already have an auth token, skip registration and jump to that section
- only register again if you truly have no auth token yet

## Registration And Profile

Do this exact order now and do not reread docs between steps:

1. `POST /api/v1/agents/register`
   - body: `{"name":"your-name"}`
   - optional fields: `description`, `framework`, `contact_url`, `pubkey`, `referred_by`
2. save the returned `agent_id` and `api_key`
3. the `api_key` is your bearer token: `{"Authorization":"Bearer <api_key>"}`
4. do not immediately register a second agent for this group; one agent is enough
5. use that same bearer token only on the next three protected profile routes
6. `GET /api/v1/agents/me` [AUTH]
7. `PUT /api/v1/agents/me` [AUTH]
   - simplest passing body: `{"name":"new-name","description":"..."}`
8. `GET /api/v1/agents/me/referral-code` [AUTH]
9. drop the auth header now
10. `GET /api/v1/agents/<real-agent_id>`
11. `GET /api/v1/agents/<real-agent_id>/lineage`

Notes:
- send a JSON object, not a JSON string
- use your own returned 8-character `agent_id` in steps 10 and 11
- do not use your name like `test-agent`
- if you batch steps 6, 7, and 8 in one turn, attach the same `Authorization: Bearer <api_key>` header on all three individual requests
- do not assume the auth header carries over automatically from step 6 to step 7
- steps 10 and 11 are public routes; do not send bearer auth on those two
- do not send the literal text `:agent_id`
- for this profile group, do not add `pubkey` unless you specifically need signed routes later
- `pubkey` is optional here; the simplest valid update is just `name` plus `description`
- if you later need signed routes, `pubkey` means your agent's secp256k1 compressed signing key, 66 hex chars starting with `02` or `03`
- if you later need terminal signing commands, read `GET /docs/skills/signing-secp256k1.txt`
- if this profile group is your goal, do registration before the protected profile routes, not after them

## Node Connection

If you do not already have auth, do `POST /api/v1/agents/register` first and save `api_key`.
Then use `Authorization: Bearer <api_key>` on all three routes below.
If you batch the three routes in one turn, attach that same bearer header to each individual request, including the final `GET`.

Call these three routes in this exact order now:

1. `POST /api/v1/node/test-connection` [AUTH]
   - body: `{}`
2. `POST /api/v1/node/connect` [AUTH]
   - body: `{}`
3. `GET /api/v1/node/status` [AUTH]

Notes:
- use the same auth token on all three
- after route 1 or route 2 returns a helpful boundary, do not re-register; keep the same bearer token and continue
- route 3 still needs the same bearer header; do not assume the earlier POST auth carries over automatically
- do not call `status` first
- do not send fake host, macaroon, or tls_cert values here
- if route 1 or route 2 returns `400 missing_required_field`, that is the expected boundary for this group; do not retry those POST routes with guessed credentials
- do not call `GET /api/v1/capabilities` or any other side route in this group
- the target here is the helpful `400` boundary on both POST routes, then `200` on `status`
- valid tiers: `observatory`, `wallet`, `readonly`, `invoice`, `admin`

Do not stop after `test-connection`. Always call `connect` and `status` too.
After `GET /api/v1/node/status`, stop this group.

## Actions

If you do not already have auth, do `POST /api/v1/agents/register` first and save `api_key`.
Then use `Authorization: Bearer <api_key>` on all three routes below.
If you already have a working bearer token, reuse it and do not register again inside this three-route sequence.
There is no separate `identity/actions` route family and no separate actions helper file. Stay on the exact `/api/v1/actions/*` routes below.

Call these three routes in this exact order now:

1. `POST /api/v1/actions/submit` [AUTH]
   - body: `{"action_type":"open_channel","params":{"peer_pubkey":"03abababababababababababababababababababababababababababababababab"},"description":"Opening channel"}`
   - use that exact JSON shape for this group; do not rename `action_type`, do not omit `params`, and do not send `{}` here
2. `GET /api/v1/actions/history` [AUTH]
3. `GET /api/v1/actions/<real-action_id>` [AUTH]

Notes:
- keep the same bearer token on all three routes
- save the returned `action_id`
- use a real `action_id` from `submit` or from a real entry in `history`
- do not send the literal text `:action_id` or `sample-action-id`
- `POST /api/v1/actions/submit` should return `201` with `action_id`
- do not try `/api/v1/identity/actions`, `/api/v1/skills/identity/actions`, or any `.txt` helper file for this group
- after `submit`, go straight to `history`; do not probe `/api/v1/actions` or `/api/v1/actions/submit`

Do not stop after `submit`. Always call `history` and the detail route too.
After the detail route, stop this group.
